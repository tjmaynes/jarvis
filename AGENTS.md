# Agents

## Project overview

Ansible project that provisions agent servers on Ubuntu Server VMs managed by [lume](https://github.com/trycua/lume) or [OrbStack](https://orbstack.dev). Supports multiple agent stacks: [Hermes Agent](https://hermes-agent.nousresearch.com/), [OpenClaw](https://openclaw.org/), and [Claude Code](https://claude.ai/code). A single playbook (`configuration/deploy.yml`) deploys to any host using `--limit`.

Roles are provided by the [`tjmaynes.agents`](https://github.com/tjmaynes/ansible-agentic-collection) Ansible Collection. This repo is a consumer of that collection.

## Key conventions

- Roles come from the `tjmaynes.agents` collection â€” installed via `make install` which runs `ansible-galaxy collection install -r configuration/requirements.yml`
- Playbooks reference roles with fully qualified names: `tjmaynes.agents.base`, `tjmaynes.agents.docker`, etc.
- The collection uses a shared `agent_name` variable to set the Linux service user, with `agent_*` prefixed variables for per-host config (e.g., `agent_discord_bot_token`, `agent_model`)
- Base role variables use the `base_*` prefix (e.g., `base_custom_dns_server`, `base_custom_ca_certificate_path`)
- Docker role variables use the `docker_*` prefix (e.g., `docker_honcho_enabled`, `docker_honcho_db_password`)
- Run `make init BACKEND=<lume|orbstack>` to bootstrap `configuration/` â€” prompts for SSH user, SSH key path (defaults to `~/.orbstack/ssh/id_ed25519` for OrbStack, `~/.ssh/id_ed25519` for lume), vault password, AI assistant (hermes/openclaw/claude-code), host name, and IP
- `configuration/` is gitignored; `configuration.example/` contains Jinja2 templates (`.j2`) rendered by `scripts/init.py` using `<< >>` / `<% %>` delimiters to avoid conflicts with Ansible's `{{ }}` syntax
- All Makefile targets that touch Ansible set `ANSIBLE_CONFIG=configuration/ansible.cfg`
- One playbook (`configuration/deploy.yml`) targets all hosts; use `--limit <host>` to deploy individually
- Hosts are defined flat under `all.hosts` in `configuration/hosts.yml` â€” no group hierarchy needed
- Per-host variables (secrets, model config, optional features) are in inventory/vault; shared config is in play-level vars
- System-level tasks run as root (`become: true` at play level)
- Optional features are conditionally enabled via `is defined` checks â€” including custom CA certs, custom DNS, GitHub tokens, Discord, Home Assistant, and Mattermost integration
- All tasks must be idempotent â€” safe to re-run without side effects
- VM creation via `scripts/setup.sh` is idempotent â€” skips if VM already exists
- Two VM backends are supported: `lume` (uses ISOs, full VM) and `orbstack` (lightweight, distro-based). Use `make setup_lume_vm` or `make setup_orbstack_vm` to create VMs, and `BACKEND=lume|orbstack` for `make start`
- OrbStack supports a `--distro` flag (default: `ubuntu`) and an optional `--amd64` flag for x86_64 emulation
- `make install` runs `brew bundle` (to install `lume` from the `Brewfile`) before creating the Python venv and installing Ansible collections
- SSH into a deployed host as a service user with `make connect_hermes HOST=<name>`, `make connect_claude HOST=<name>`, or `make connect_openclaw HOST=<name>` (opens a tmux session)

## Secrets

Sensitive values use `no_log: true` and file mode `0600`. Always run `make encrypt` after editing `configuration/vault.yml`.

This repo is public â€” never commit plaintext secrets. Host IPs, tokens, and auth keys go in the encrypted vault inside `configuration/`.
